# ChunkBox

**ChunkBox** یک دانلودر ساده بر پایه **GitHub Actions** است که لینک مستقیم فایل را دریافت می‌کند، فایل را دانلود می‌کند، در صورت نیاز آن را به پارت‌های کوچک‌تر تقسیم می‌کند و خروجی را داخل ریپو یا GitHub Release ذخیره می‌کند.

این پروژه برای شرایطی ساخته شده که دسترسی مستقیم به بعضی منابع، سرورها یا فایل‌ها برای کاربران داخل ایران محدود، ناپایدار یا مسدود است. ChunkBox تلاش می‌کند با استفاده از زیرساخت GitHub Actions یک راه ساده برای دریافت، تقسیم و نگهداری فایل‌ها فراهم کند؛ بدون نیاز به سرور شخصی.

> از این ابزار فقط برای فایل‌هایی استفاده کنید که اجازه دانلود و نگهداری آن‌ها را دارید.

---

## قابلیت‌ها

- دانلود فایل از لینک مستقیم
- پشتیبانی از یک یا چند لینک همزمان
- تقسیم خودکار فایل‌های بزرگ به پارت‌های کوچک‌تر
- ذخیره خروجی در دو حالت:
  - داخل ریپو
  - داخل GitHub Release
- ساخت فایل `manifest.json` برای هر دانلود
- ساخت فایل SHA256 برای بررسی سلامت فایل
- ساخت اسکریپت `rebuild.sh` برای اتصال دوباره پارت‌ها
- ساخت گزارش نهایی در GitHub Actions Summary
- پشتیبانی از لینک‌های خصوصی با GitHub Secrets

---

## ساختار پروژه

```text
.github/
  workflows/
    downloader.yml

README.md
```

در حالت ذخیره داخل ریپو، خروجی‌ها معمولاً در پوشه `downloads` قرار می‌گیرند:

```text
downloads/
  2026-05-01_113525_001_file_ab12cd34/
    file.zip.part-0000
    file.zip.part-0001
    file.zip.sha256
    rebuild.sh
    README.md
    manifest.json
```

---

## نصب

فایل Workflow را در این مسیر قرار دهید:

```text
.github/workflows/downloader.yml
```

بعد از commit و push، وارد تب **Actions** در GitHub شوید و Workflow با نام **ChunkBox Downloader** را اجرا کنید.

## نحوه استفاده

در صفحه اجرای Workflow، مقادیر مورد نیاز را وارد کنید.

### `urls`

لینک یا لینک‌های مستقیم دانلود.

برای یک لینک:

```text
https://example.com/file.zip
```

برای چند لینک:

```text
https://example.com/file1.zip
https://example.com/file2.iso
https://example.com/file3.mp4
```

---

### `storage_mode`

محل ذخیره خروجی را مشخص می‌کند.

```text
repo
release
auto
```

- `repo`: فایل‌ها داخل ریپو ذخیره و commit می‌شوند.
- `release`: فایل‌ها به GitHub Release آپلود می‌شوند.
- `auto`: برای فایل‌های کوچک از ریپو و برای فایل‌های بزرگ از Release استفاده می‌شود.

پیشنهاد برای فایل‌های بزرگ:

```text
storage_mode: auto
```

---

### `output_folder`

نام پوشه خروجی در حالت `repo`.

```text
downloads
```

---

### `split_size_mib`

اندازه هر پارت بر حسب MiB.

مقدار پیشنهادی:

```text
98
```

این مقدار باعث می‌شود هر پارت زیر محدودیت ۱۰۰ مگابایت GitHub باقی بماند.

---

### `repo_push_batch_mib`

حداکثر حجم هر batch هنگام push در حالت `repo`.

مقدار پیشنهادی:

```text
500
```

این گزینه کمک می‌کند فایل‌های بزرگ به‌جای یک push سنگین، در چند push کوچک‌تر ارسال شوند.

---

### `release_tag`

در حالت `release` یا `auto`، نام tag برای Release را مشخص می‌کند.

مثال:

```text
pixelos-miatoll
```

اگر خالی بماند، ChunkBox خودش یک tag می‌سازد.

---

## پیشنهاد استفاده

برای فایل‌های معمولی:

```text
storage_mode: repo
split_size_mib: 98
repo_push_batch_mib: 500
```

برای فایل‌های بزرگ:

```text
storage_mode: auto
split_size_mib: 98
repo_push_batch_mib: 500
```

برای اینکه ریپو سنگین نشود:

```text
storage_mode: release
split_size_mib: 98
```

## بازیابی فایل‌های چندپارتی

اگر فایل به چند پارت تقسیم شده باشد، کنار پارت‌ها یک فایل `rebuild.sh` ساخته می‌شود.

برای اتصال دوباره پارت‌ها:

```bash
bash rebuild.sh
```

یا به‌صورت دستی:

```bash
cat file.zip.part-* > file.zip
sha256sum -c file.zip.sha256
```

اگر خروجی `OK` باشد، فایل سالم بازسازی شده است.

## فایل manifest.json

برای هر دانلود، یک فایل `manifest.json` ساخته می‌شود که اطلاعات اصلی فایل را نگه می‌دارد:

```json
{
  "url": "https://example.com/file.zip",
  "filename": "file.zip",
  "size_bytes": 2453667840,
  "sha256": "abc123...",
  "split": true,
  "part_count": 26,
  "storage_mode": "repo"
}
```

این فایل برای بررسی، مستندسازی و جلوگیری از دانلودهای تکراری کاربرد دارد.

## لینک‌های خصوصی

اگر لینک دانلود نیاز به Authorization header داشته باشد، مقدار آن را داخل GitHub Secrets ذخیره کنید.

مسیر:

```text
Settings > Secrets and variables > Actions > New repository secret
```

نام Secret:

```text
DOWNLOAD_AUTH_HEADER
```

نمونه مقدار:

```text
Bearer YOUR_TOKEN
```

یا:

```text
Authorization: Bearer YOUR_TOKEN
```

توکن‌ها را هیچ‌وقت داخل فایل‌های پروژه commit نکنید.

## محدودیت‌ها

ChunkBox برای لینک‌های مستقیم طراحی شده است. لینک‌هایی که نیاز به ورود، کوکی، کپچا یا کلیک در مرورگر دارند ممکن است مستقیم کار نکنند.

نمونه سرویس‌هایی که ممکن است نیاز به لینک مستقیم واقعی داشته باشند:

- Google Drive
- Mega
- MediaFire
- Dropbox با صفحه واسط
- لینک‌های دارای کپچا
- لینک‌های نیازمند کوکی

---

## نکات مهم

- برای فایل‌های بزرگ، حالت `release` یا `auto` بهتر از `repo` است.
- ذخیره فایل‌های حجیم داخل ریپو باعث سنگین شدن history می‌شود.
- هر پارت بهتر است کمتر از ۱۰۰ مگابایت باشد.
- اگر Workflow وسط کار متوقف شود، ممکن است فقط بخشی از پارت‌ها ذخیره شده باشند.
- فایل‌های حساس، خصوصی یا بدون مجوز را با این ابزار دانلود و منتشر نکنید.
